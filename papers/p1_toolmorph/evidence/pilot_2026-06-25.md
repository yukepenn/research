# P1 ToolMorph — pilot evidence (2026-06-25)

Real models via subscription CLIs (no API cost): `claude` (Claude Code) and `codex` (gpt-5.5).
Unified interactive plan harness (`p1_interactive_v1`): the model sees each tool result before
the next call; we execute against the deterministic env + invariant oracle. All episodes in the
immutable ledger (`artifacts/run_ledger.jsonl`, paper_id=p1_toolmorph).

Design: 10 base tasks (calendar/inventory/helpdesk) × {Claude, Codex} × {original,
structural_nesting, enum_encoding, lexical_aliasing}. 79 valid episodes (1 infra error excluded).

## Result — paired success vs the original interface
| interface | Claude success | Δ vs original | Codex success | Δ vs original |
|---|---|---|---|---|
| original | 1.00 | — | 1.00 | — |
| structural_nesting | 1.00 | 0.00 | 1.00 | 0.00 |
| lexical_aliasing | 1.00 | 0.00 | 1.00 | 0.00 |
| **enum_encoding** | **0.80** | **−0.20** | **0.89** | **−0.11** |

## Reading (honest)
- With the single-shot id-prediction confound removed (baseline = 1.00), modern strong models are
  ROBUST to structural nesting and lexical aliasing on these tasks.
- **Both models degrade under enum_encoding** (string enum values replaced by opaque short codes) —
  a state-transition-equivalent change. Directionally consistent across two independent families.
- Effect is modest and tasks are easy, so CIs are wide (n≈10 tasks/cell); this is PILOT signal
  toward H1, not a confirmatory result. It motivates: (a) harder/more tasks, (b) more transforms
  and enum variants, (c) the load-bearing test — does label-free LLM-STNF RECOVER enum performance
  on held-out tools? (static canonicalizer provably cannot; this is the surviving novel core).
