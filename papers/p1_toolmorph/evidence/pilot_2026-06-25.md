# P1 ToolMorph — pilot evidence (2026-06-25)

Real models via subscription CLIs (no API cost): `claude` (Claude Code) and `codex` (gpt-5.5).
Unified interactive plan harness (`p1_interactive_v1`): the model sees each tool result before
the next call; we execute against the deterministic env + invariant oracle. All episodes in the
immutable ledger (`artifacts/run_ledger.jsonl`, paper_id=p1_toolmorph).

Design: 10 base tasks (calendar/inventory/helpdesk) × {Claude, Codex} × {original,
structural_nesting, enum_encoding, lexical_aliasing}. **80 episodes, none excluded.** The single
Claude/enum failure is a max_steps TIMEOUT (the agent looped under opaque codes) — a genuine agent
failure, COUNTED, not excluded (see analysis_plan missing_data_policy).

## Reading (honest — corrected after red-team)
- With the single-shot id-prediction confound removed (baseline = 1.00), modern strong models show
  NO observed drop under structural nesting or lexical aliasing — though the 1.00 ceiling means
  absence-of-effect cannot be distinguished from insufficient power.
- Under enum_encoding (string enum values replaced by opaque short codes — a state-transition-
  equivalent change) both models show a SMALL drop (Claude −0.20, Codex −0.10), **directionally
  consistent across two families but NOT statistically distinguishable from zero: both bootstrap CIs
  include 0**. This is a directional PILOT signal toward H1, not a confirmed effect; it rests on 1–2
  failing episodes per model at a 1.00 ceiling with a single draw per cell.
- It motivates the decisive, still-unrun experiments: (a) harder tasks to escape the ceiling +
  scale to 24–120 tasks with repeats/seeds, (b) the H3 content/legibility-matched control (rebut
  Schema First), (c) the load-bearing PRIMARY test — does label-free LLM-STNF RECOVER enum
  performance on held-out transforms/tools? (the static canonicalizer provably abstains on enum;
  this remedy, H5, is the surviving novel core and is currently UNRUN — a pre-registered plan).
